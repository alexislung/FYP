8. Critical Evaluation

Critical Evaluation of Core Function: AI Career Coach

This section evaluates the AI-powered Career Coach — the part of the system that uses DeepSeek to give personalized career guidance in real time.

Technical Implementation & Performance

Streaming responses.
The biggest technical change in this module was moving to Server-Sent Events (SSE). At first, the system used a normal request–response cycle. Because LLM generation takes time, users were often left waiting 5–10 seconds with a frozen interface. I changed the Flask backend so it streams instead: the DeepSeek API is called with `stream=True`, and a Python generator pushes chunks to the client as they arrive. That brought the “time to first byte” down to under 500ms. The response now appears gradually, like a typewriter, instead of all at once, which makes the experience feel much more responsive.

[Additional] Upstream failures are surfaced when the API returns a non-success status; in that case the backend responds with JSON rather than a stream, so the client can show an error instead of failing silently. After the stream completes, the assistant’s full reply is persisted so the conversation can be retrieved later via the history endpoint.

Context management.
The system keeps the last 10 messages in a PostgreSQL-backed context window, so the model can follow the conversation (e.g. that the user is interested in “Marketing” roles). This works well for short-to-medium chats. Once the conversation gets longer, though, token count goes up, which increases latency and API cost. A next step would be something like a sliding window or a summary of older turns to keep context smaller.

[Additional] In deployment, chat history is stored in PostgreSQL (accessed via Supabase). A production-oriented improvement would be per-user isolation of messages and explicit retention rules, because a shared global message table in a prototype can mix sessions and raises privacy questions if the product scales beyond a demo.

 User Experience (UX) and Interface Design

Natural language interaction.
 Unlike keyword search, users can ask broad questions like “I’m good at talking to people, what job fits me?” and the system still maps that to categories such as Sales or Customer Service. The downside is that the current UI is text-only. There’s no resume upload (PDF/Word) for direct analysis in this version. Adding file parsing would be a big improvement for personalization.

Visual feedback.
The typing animation and auto-scroll keep attention on the latest message. When the network is slow or congested, the stream can stutter, so text sometimes arrives in larger chunks instead of smoothly. It’s still usable, but noticeable.


 AI Reliability and Hallucination Risks

Domain knowledge
 DeepSeek is strong on general career advice (e.g. the STAR method for interviews). The main weakness is that it doesn’t know Hong Kong–specific job market details by default. For example, it might quote salary ranges that fit the US market but not Hong Kong. I added a system prompt telling the model to act as a “helpful career assistant for the Hong Kong job market” and to prioritise local context. That cut down a lot of off-target advice — rough testing suggests maybe around 70% — but it’s not perfect, and manual checks are still a good idea.

[Additional] The “~70%” figure is not a formal benchmark; it reflects informal manual review of a small prompt set. A stronger evaluation would fix explicit criteria (e.g., salary plausibility for HK, relevance to local industries) and a larger sample.

Liability
The tool is meant as an advisor, not a certified counsellor. There’s a risk that someone might make major career decisions purely on AI output. I added a disclaimer in the UI: “AI advice is for reference only.” The goal is to frame it as support for human judgment, not a substitute for it.

[Additional] Users should treat coaching output as non-confidential technical guidance unless the deployment adds proper access control and data policies; this is especially important if career concerns overlap with health or legal matters, where referral to a qualified professional is appropriate.


Critical Evaluation of Core Function: Job Search

This section evaluates the Job Search module — the part of the system where users filter opportunities by location, salary, and job type, view trending and recommended listings, and use their CV to receive job matches tailored to their skills and experience.

Problems and Difficulties Encountered

CV parsing and format inconsistency
 For CV-based matching, the system must extract skills, experience, and education from the CV. When the CV comes from the CV Editor, the data is structured and easy to use. When users upload a PDF or Word file, parsing becomes harder: layouts vary, languages can mix (English, Chinese), and formats differ. Building or integrating a robust CV parser that handles diverse formats adds complexity.

[Additional] Scope of CV matching: personalised ranking is strongest when the CV yields clean structured features (skills, titles, years of experience). Unstructured uploads increase noise; match scores should be interpreted as indicative, and users should cross-check shortlisted roles against the original job descriptions.

Designing the matching algorithm
 Deciding how to score “fit” between a CV and a job is difficult. Keyword matching is simple but misses semantic similarity (e.g. “project management” vs “managing projects”). Semantic matching via an LLM is more nuanced but introduces latency, API cost, and potential inconsistency. Balancing relevance (skills, experience level, industry) with user filters (location, salary, type) and ranking results in a sensible order requires careful tuning.

[Additional] Compared with large commercial aggregators (e.g. JobsDB, LinkedIn Jobs), our prototype does not aim to match their listing volume. The contribution is an integrated student workflow (filters + CV-informed discovery) within our stack; freshness and breadth of listings remain dependent on how data is sourced and maintained.

Potential Difficulties

Data freshness and coverage
 Even with a database, job listings must be updated regularly. Stale or sparse data will hurt user trust. Sourcing, cleaning, and maintaining Hong Kong job data at scale is non-trivial.

 Limitations
The “Scope of CV matching” subsection now covers how CV-based matching adds personalization and what its effectiveness depends on.

[Additional] Where the apply flow uses simulated or heuristic scoring for demonstration, results illustrate the UX path rather than a validated ML ranker; extending this to reproducible metrics would be a natural next step.


Critical Evaluation of Core Function: Job Quiz

This section evaluates the Job Quiz — the part of the system where users answer 35 career-related questions and receive an AI-generated report on personality, ability, education, and working conditions.

Problems and Difficulties Encountered

 The challenge of Designing the questions
 The Quiz had to cover personality, ability, education, and working conditions in a way that was clear to users to choose and useful for the AI to generate the results. We extended the quiz from 10 to 35 questions to support a proper career report, which required careful wording and consistent answer options. 
Also, understanding the user’s personality, skills, and related aspects is inherently limited: the system only knows what users choose in fixed options, so it cannot capture nuance, contradictions, or context that a real interview would. The quality of suggestions and analysis in the report depended heavily on how the AI was prompted and how quiz data was sent. We had to refine the n8n workflow and the DeepSeek prompt so the model received a clear JSON payload and produced a concise 300–400 word report instead of generic or overly long text. 

[Additional] The quiz should be read as exploratory self-report guidance, not a substitute for validated psychometric instruments.

 System linkage and technical issues
The HTML quiz runs in the browser and sends data to an n8n webhook. CORS had to be configured on the webhook, and testing from a local file failed until the site was served over HTTP. We also had to fix the webhook response format so the front end could show the report and pass it to a separate report page for viewing and PDF download.

[Additional] The Flask backend also exposes a server-side forwarding route for quiz analysis, which avoids browser CORS issues when calling n8n and allows a request timeout to be enforced so the UI does not hang indefinitely if the workflow is slow.

[Additional] Report export uses client-side PDF generation on the report page; very long outputs or browser rendering differences can affect layout, so “Print to PDF” remains a useful fallback.

[Additional] Quiz generation depends on the availability of the n8n workflow; outages or workflow edits can interrupt report delivery until the integration is updated.


Critical Evaluation of Core Function: CV Temples

This section evaluates the CV Template  — the part of the system where users pick a template, fill in their details across several steps, and use AI to generate a CV and cover letter.


Problems and Difficulties Encountered

Selecting and designing templates
mean balancing layout, readability, and compatibility with different screen sizes and with PDF generation. Offering several templates (e.g. single-column, two-column) required a consistent structure so that the same data could be reused across templates. 

Generation vs. checking
The AI generates or rewrites content based on the submitted data. If the user enters sparse or messy data, the output can be generic or inconsistent. We validate required fields (e.g. name, email) before sending, but the AI has no way to verify facts or fill gaps.

Integration with n8n and DeepSeek
We originally planned to use OpenAI, but it’s not available in Hong Kong, so we switched to DeepSeek via n8n. The webhook workflow takes the form data, builds a prompt, calls the DeepSeek Chat node, and returns the generated text. 

[Additional] Aligning structured CV fields from the editor with downstream job matching reduces ambiguity compared with free-text-only CVs; any gap between editor fields and uploaded files remains a practical limitation.


Potential Difficulties

Accuracy and fabrication
The AI might add details that weren’t in the user’s input or phrase things in a way that oversells qualifications. Users should review the generated CV and cover letter before sending. A note in the UI reminding them to verify content would be useful.


Delays and Changes in Project Schedule

Strengthening team communication and mutual progress supervision

 We have less information about our process and also some communication difficulty between groupmates. So strengthening team communication and mutual progress supervision would have helped surface these integration and environment dependencies earlier and reduce rework.

[Additional] Concrete examples of late-discovered dependencies in our project included webhook response formats, serving the site over HTTP for integration testing, and coordinating AI prompts across modules (coach, quiz, CV). Earlier end-to-end integration tests and a short weekly sync on interfaces would likely have reduced rework.


Reflection on the team’s use of generative AI

We used generative AI to support drafting, debugging, and prompt iteration. Architectural decisions, API wiring, database behaviour, and checking of user-visible claims remained the team’s responsibility. AI-generated coaching and CV text was treated as a draft: prompts and structured payloads (e.g. JSON to n8n) were refined to constrain tone and format, and outputs were reviewed for plausibility and inappropriate claims before being presented as final guidance.


Overall conclusion

Overall, the prototype integrates conversational coaching (with streaming responses), job discovery, quiz-based reporting, and CV/cover-letter generation. The main strengths are perceived responsiveness from SSE and a single workflow for students exploring careers in Hong Kong. The main limitations are Hong Kong–specific factual grounding without a curated knowledge base, job data freshness and coverage, heterogeneous CV uploads, and the inherent limits of fixed-option quiz items. Dependence on external LLM APIs implies latency, cost, and availability trade-offs. If additional time were available, priorities would be: (1) stronger evaluation metrics for matching and advice quality, (2) more robust parsing or stronger structured CV flows, and (3) improved data maintenance for listings.
